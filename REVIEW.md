# 🔍 Audit du code — erreurs trouvées & améliorations appliquées

Audit complet réalisé sur l'API Laravel et l'app mobile après la première version.
Voici les problèmes identifiés et les corrections/améliorations appliquées dans cette v2.

## ✅ Erreurs / incohérences corrigées

| # | Problème | Sévérité | Correction |
|---|----------|----------|------------|
| 1 | **Création d'un produit avec quantité initiale** : la quantité était enregistrée directement, sans mouvement → l'historique et le graphique dashboard étaient incohérents avec le stock réel | 🟠 Moyen | Un mouvement d'entrée **« Stock initial »** est désormais créé en transaction (`ProductController@store`) |
| 2 | **Double-tap / retry réseau sur l'app** : deux clics rapides sur « Enregistrer » créaient deux mouvements identiques | 🟠 Moyen | **`client_uuid` unique** sur `stock_movements` : le serveur rejoue la requête en toute sécurité (idempotence) — indispensable aussi pour le mode hors ligne |
| 3 | **Produits non consultables hors ligne** : l'app affichait juste une erreur réseau | 🟠 Moyen | **Cache AsyncStorage** de la liste produits + repli automatique en cas d'échec réseau (badge « données en cache ») |
| 4 | **Fiche produit inaccessible hors ligne** même si le produit était dans le cache de la liste | 🟡 Faible | Le produit complet est transmis via les params de navigation → la fiche s'affiche avec un bandeau « pas à jour » |
| 5 | **Pas de photo produit** alors que le sujet « vente » l'exige | 🟠 Moyen | Colonne `image_path`, upload multipart validé (`image/jpg,png,webp ≤ 3 Mo`), suppression de l'ancienne photo, URL publique `asset('storage/...')` |
| 6 | **Dashboard lent en croissance** : requêtes par date sans index dédié | 🟡 Faible | Index `created_at` ajouté sur `stock_movements` |
| 7 | **Ancienne photo jamais supprimée** du disque lors d'un remplacement → disque qui grossit | 🟡 Faible | `Storage::disk('public')->delete()` de l'ancien fichier avant remplacement |
| 8 | **UX mobile** : thème clair peu valorisant, incohérences de contrastes | 🟡 Faible | **Thème sombre premium** complet (palette violet/cyan assortie au site), `#DBEAFE` illisible remplacé, header/tab bar harmonisés |

## 📝 Limites connues (choix assumés, documentés)

- Les mouvements créés **hors ligne** obtiennent leur date serveur à la synchronisation (la date locale est conservée dans la file, non propagée) — acceptable pour la compta simplifiée
- Si un produit est **supprimé ou renommé** pendant que l'app est hors ligne, les mouvements en attente échouent à la sync et restent en file avec leur erreur (décision manuelle)
- Le stock n'est **pas décrémenté localement** hors ligne (évite de fausser le stock réel) ; le contrôle « stock insuffisant » s'applique au moment de la sync
- Les prix de vente < prix d'achat ne sont pas bloqués (peut être une promo) — simple champ à ajouter si besoin
- ~~**App** : tous les écrans ne sont pas traduits~~ → **résolu en v4 : 15/15 écrans FR/EN** ✅
- **Instructions de paiement** : stockées en une seule langue (celle saisie par l'admin, FR par défaut) — pour du bilingue, ajouter une colonne `instructions_en`
- **Emails** : envoyés en français ; pour localiser, passer par `__('...')` avec la locale de l'acheteur
- **Push distantes** : fonctionnent en **dev build / APK** avec `npx eas init` (projectId) ; dans **Expo Go** (SDK 53+) le token distant est indisponible → repli silencieux sur les notifications **locales** planifiées à l'ouverture de l'onglet Admin
- **Textes des push** : envoyés en français par le serveur (comme les emails) — une localisation par destinataire nécessiterait un champ `locale` sur `users`
- **Impression Bluetooth** : nécessite `npx expo prebuild` + dev build EAS (`react-native-thermal-receipt-printer`) ; dans Expo Go tout le reste fonctionne, le bouton explique la marche à suivre. Les accents sont volontairement retirés des tickets BT (compatibilité codepage imprimante). Imprimantes 58 mm : ajuster `WIDTH` (42 → 32) dans `utils/thermalPrinter.js`
- **PO** : seules les **quantités** sont éditables (pas le prix unitaire) — snapshot du prix d'achat au moment de la génération, par design
- **PDF** : nécessitent `barryvdh/laravel-dompdf` installé ; sans lui, les emails partent sans pièce jointe et les routes `/recu` renvoient une erreur propre

## ✅ v14 — En transit 🚚 · Réception validée ✓ · Alertes par emplacement 📍

### Stock pendant le transit — décision comptable clé
À l'envoi : `addDelta(source, −qty)` → bucket **et global** diminuent. À la réception : `addDelta(destination, +qty)` → global remonte. Invariant préservé : **global = siège + Σ(boutiques) à tout instant** ; le stock sur la route ne pollue ni le siège ni une boutique (défaut identifié de l'approche « global inchangé » : le siège aurait pu vendre le contenu du camion).
- Receive/cancel verrouillent le transfert (`lockForUpdate`) + garde `isPending()` (anti double-tap / course réseau).
- Annulation : remet le stock à la source + mouvement `transfer_in` « Annulation du transfert ».
- Permissions : réception = admin OU manager de la destination ; annulation = admin OU manager de la source (checks en contrôleur, pas seulement middleware).

### Alertes par emplacement (`GET /api/stock-alerts`, staff)
Résolution : `?shop_id=` (admin / manager sur sa boutique) > boutique du user > siège. Seuil = `alert_threshold` produit (global, pas de seuil par boutique — documenté).
